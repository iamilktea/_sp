
# 系統程式設計期中專案報告：AuraShell 實作解析

**專案名稱**：AuraShell (簡易 Linux 命令列直譯器)  
**開發語言**：C Language (C99)  
**涵蓋技術**：行程控制 (Process Control)、並行管理、檔案系統 I/O 重導向  

## 一、 專案簡介與動機
在本次「系統程式」課程中，我們深入探討了作業系統如何管理硬體資源與應用程式。為了將所學的抽象概念（如 `fork`, `exec`, `wait`）具象化，本專案從零實作了一個微型的 Linux Shell——**AuraShell**。

AuraShell 具備以下核心能力：
1. **標準指令執行**：利用作業系統的 `exec` 家族函式載入並執行外部二進位檔。
2. **I/O 重導向 (`>`)**：實作檔案描述子 (File Descriptor) 的切換，將程式輸出寫入檔案。
3. **背景並行執行 (`&`)**：實作非阻塞的行程等待機制，讓 Shell 能與長時間執行的子行程並行運作。

## 二、 系統架構與核心 System Calls
本系統高度依賴 POSIX 標準的系統呼叫，其核心生命週期與架構如下：

### 1. 細胞分裂與大腦移植 (Fork-Exec 模式)
當使用者輸入指令時，AuraShell 不會親自執行它，而是：
* **`fork()`**：產生一個與自身相同的子行程。這確保了即便使用者輸入錯誤的指令導致崩潰，也只有子行程會死掉，AuraShell 本體依然存活。
* **`execvp()`**：在子行程內部呼叫此函式，將記憶體空間替換為目標程式（如 `ls`）。

### 2. 檔案登記表的魔術 (dup2)
當解析到 `>` 符號時，AuraShell 會在子行程中觸發 I/O 重導向機制：
* **`open()`**：以 `O_WRONLY | O_CREAT | O_TRUNC` 模式開啟目標檔案，並取得新的 FD（例如 3）。
* **`dup2(fd, STDOUT_FILENO)`**：將標準輸出（FD 1，預設為螢幕）強制指向剛剛開啟的檔案。自此，子行程內所有的 `printf` 都會寫入檔案中。

### 3. 孤兒與殭屍行程管理 (Zombie Process Prevention)
當指令尾隨 `&` 時，父行程不會呼叫阻塞的 `waitpid`。但若子行程結束而父行程不回收它，就會產生佔用系統資源的「殭屍行程」。
* **解法**：在 AuraShell 的主迴圈開頭，加入 `while (waitpid(-1, NULL, WNOHANG) > 0);`。這是一段非阻塞的清理程式，能默默在背景回收所有已終止的子行程，維持系統整潔。

## 三、 開發挑戰與解決方案
**挑戰：字串解析與記憶體破壞**
在早期實作中，使用 `strtok` 切割指令字串時，若沒有正確將最後一個參數設為 `NULL`，會導致 `execvp` 發生 Segmentation Fault (記憶體區段錯誤)。
**解決方案**：在 `ParseCommand` 函式中，嚴格追蹤陣列索引，並在迴圈結束時強制指派 `args[i] = NULL;`，確保符合底層 API 的記憶體邊界要求。

## 四、 學習心得與結論
透過實作 AuraShell，我深刻體會到「萬物皆檔案」的 UNIX 哲學。以往只會使用 Linux 終端機，現在終於理解每一次敲擊 Enter 鍵背後，作業系統是如何優雅地進行行程複製、記憶體替換與資源回收。

本次專案不僅將課程前半段的「C 語言指標」、「檔案系統操作」與「行程並行控制」完美融合，也為未來進階的伺服器開發（如 Socket Programming 與 Multithreading）打下了極為堅實的底層基礎。
